# LeadOps Studio bot runbook

## Назначение

MVP Telegram-бота для inbound CTA лендинга:

- принимает пользователя из CTA;
- проводит 6 вопросов квалификации;
- считает простой скоринг;
- сохраняет лид локально;
- отправляет карточку лида менеджеру.

## Секреты

Файл `.env` не коммитится.

Обязательные переменные:

```bash
LEADOPS_BOT_TOKEN=<token from BotFather>
LEADOPS_MANAGER_CHAT_ID=<telegram chat id для карточек лидов>
```

Важно: token из BotFather обычно выглядит как `<digits>:<secret>`. Если `getMe` возвращает 404 `Not Found`, почти всегда token скопирован не полностью или уже отозван.

## Проверка токена

```bash
./scripts/run-bot.sh --check
```

Ожидаемый ответ:

```json
{"ok": true, "username": "...", "first_name": "..."}
```

## Зарегистрировать команды

```bash
./scripts/run-bot.sh --set-commands
```

Команды:

- `/start` — начать разбор заявки
- `/demo` — собрать demo-сценарий
- `/audit` — разобрать обработку заявок
- `/pilot` — узнать про пилот

## Запуск локально

```bash
./scripts/run-bot.sh
```

Для фонового запуска на сервере:

```bash
nohup ./scripts/run-bot.sh > data/bot.log 2>&1 &
```

## Локальные данные

Не коммитить:

- `data/bot-state.json`
- `data/leads.jsonl`
- `data/bot.log`

Они могут содержать персональные данные лидов.

## Тестовый сценарий

1. Открыть Telegram-бота.
2. Нажать Start.
3. Нажать “Собрать demo-сценарий”.
4. Пройти 6 вопросов.
5. Проверить, что менеджеру пришла карточка лида.

Для отправки карточек менеджеру этот менеджер должен хотя бы раз нажать Start в боте, иначе Telegram не даст боту писать первым.

Рекомендация для продакшена: использовать отдельный закрытый manager chat/group для служебных карточек. Если тестирующий клиентский chat_id совпадает с `LEADOPS_MANAGER_CHAT_ID`, бот не отправляет внутреннюю карточку в клиентский диалог, чтобы клиент не видел служебную информацию.
